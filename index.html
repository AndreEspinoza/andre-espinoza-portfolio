Artifacts and Enhancements

For this capstone I focused on one main artifact, the Travlr Getaways web application from CS 465. It is a MEAN stack project that started as a basic travel booking site and then became the core place where I applied three different enhancement paths: software engineering and design, algorithms and data structures, and databases.

Software Engineering and Design

Originally, the backend exposed public CRUD operations on trips with minimal validation. I implemented JSON Web Token authentication and role-based access control so only admins can modify trip data.

On the server, requireAuth verifies the JWT and requireRole('admin') protects POST, PUT, and DELETE routes for trips. Missing or invalid tokens return a 401 response, while valid tokens with insufficient privileges return a 403 forbidden response.

On the Angular client, an HTTP interceptor automatically attaches the Bearer token to API requests. An AuthGuard enforces role-based routing, and the UI hides admin-only actions unless the authenticated user has the admin role. This separation of concerns mirrors real-world production patterns and keeps the codebase secure and maintainable.

Algorithms and Data Structures

In the original version, trips were returned in an unranked list. I added a simple recommendation algorithm that scores trips based on user preferences such as budget, continent, category, and desired trip length.

Each trip is assigned a score, and the list is sorted using an O(n log n) sort. Trips that best match the user’s preferences are displayed first. This approach balances performance, clarity, and explainability without overengineering the solution.

This enhancement demonstrates algorithmic thinking, trade-off analysis, and the ability to design efficient solutions appropriate to the scale of the application.

Databases

The database enhancement focused on adding a booking system with capacity enforcement. Trips now include a capacity field, and bookings link users to specific trips.

The booking flow uses a single atomic findOneAndUpdate operation that checks available capacity and decrements it in one step. Successful bookings return a 201 response with updated capacity, while overbooking attempts return a 409 conflict without modifying data.

This design protects data integrity, prevents race conditions, and demonstrates proper use of database constraints in concurrent environments.
